## PART 1 — "CODEBASE" STANDARDS FOR AI AGENTS (RESEARCH SYNTHESIS)

### 1.1. What is Antigravity IDE (July 2026 Context)

Antigravity is Google's AI coding agent / IDE. Unlike Claude Code (which runs in CLI/agentic mode), Antigravity is a full IDE with a project-aware agent—it reads the entire workspace, understands the structure, and performs multi-file edits.

**Implications for the codebase**: Since the agent reads the *entire* workspace for context, how you organize files/directories and write rule documents (`AGENTS.md` / `CLAUDE.md` / memory) directly determines the quality of the code generated by the agent.

### 1.2. General Rules for All AI Agents (Cross-tool: Antigravity, Cursor, Windsurf, Claude Code, Cline)

Research from Martin Fowler, Simon Willison, IEEE Spectrum, and ACM Queue (2025-2026) indicates:

| Principle | Rationale | Source |
|---|---|---|
| **Write clear `AGENTS.md` / `CLAUDE.md` with hard constraints** | Agents lacking rules generate freestyle code → technical debt accumulates quickly | [Martin Fowler - Responsible AI-Assisted Dev] |
| **Consistent, predictable directory structure** | Agents rely on conventions to guess file locations. Messy folders → misplaced files | Best practices summary (Cursor/Windsurf/Claude) |
| **Clear separation of logic ↔ presentation (MVVM/MVC)** | Tightly coupled projects (like QML/C++) need clear constraints so agents don't shove logic into the wrong layers | [The Register - Vibe Coding Tech Debt] |
| **Always have a verify/build/test step before "done"** | ~30% of AI-generated codebases have at least 1 vulnerability; devs often accept untested code | [ACM Queue - Over-Reliance on AI Code] |
| **Transparent task tracking (`tasks_board`, todo)** | Agents need to know "what phase we are in" to avoid repeating or skipping tasks | Simon Willison - Vibe Coding Future |
| **Context hygiene (`.gitignore` / ignore files)** | Remove build artifacts and media from context so the agent can focus | Best practices (Claude Code memory) |

**Important Warning (Contrarian View)**: Vibe coding is a *prototype accelerator*, not a *production methodology*. Research emphasizes: always incorporate code review + testing + architectural analysis—exactly what your project has already set up (`qt-qml-review`, `qt-cpp-review` skills).

### 1.3. Specific Qt6/QML Standards (From Official Qt + Your Project's Skill System)

Your project already has **Qt Company official skills** (`qt-qml`, `qt-cmake-project`, `qt-cpp-review`, `qt-qml-review`)—these are the most practical best practices, requiring no further research. Summary of hard rules:

1. **Build system**: Use `CMake` + `qt_standard_project_setup()`. Do not use qmake (`.pro`).
2. **QML module**: Every `.qml` must go through `qt_add_qml_module()`—NO hand-written `.qrc` for QML.
3. **C++ ↔ QML**: Use `QML_ELEMENT` / `Q_PROPERTY` / `Q_INVOKABLE`, no manual `qmlRegisterType`.
4. **Qt 6 Imports**: No version numbers, no `QtQuick.Window` if `QtQuick` is already imported.
5. **File naming**: PascalCase (`MyButton.qml`), URI matching the directory path.
6. **Layout**: Don't mix `anchors` + `Layout.*`. Use `Layout.*` for sizing within Layouts.

Current `CMakeLists.txt`:

**Violations of `qt-cmake-project` hard rules:**
- ❌ Rule 3: `find_package(Qt6 REQUIRED ...)` lacks a minimum version → should be `find_package(Qt6 6.8 REQUIRED ...)`
- ❌ Rule 2: Hand-rolled `set(CMAKE_AUTOMOC ON)` instead of letting `qt_standard_project_setup()` handle it (skill says: "Do not manually set CMAKE_AUTOMOC/AUTOUIC").
- ❌ Rule 1: Uses `qt_standard_project_setup(REQUIRES 6.5)` but the skill recommends `REQUIRES 6.8` (due to QTP0005 policy for TARGET imports requiring 6.8+).
- ❌ Custom Rule 1 (Linux Mint): Missing `CMAKE_BUILD_TYPE` guard, `CMAKE_EXPORT_COMPILE_COMMANDS ON`, `-Wall -Wextra`, and Ninja generator.
- ⚠️ `resources/qml.qrc` is a hand-written `.qrc`—the skill explicitly says "No hand-written .qrc for QML" (QML should go via `qt_add_qml_module`, assets via `RESOURCES` arg).

**Consequence**: The build might succeed (Qt Creator configures successfully), but it violates the project's own best practices → future agents will copy this flawed pattern.

#### ❌ ERROR #2: `main.cpp` uses `QUrl("qrc:/...")` instead of `loadFromModule()`

```cpp
const QUrl url(QStringLiteral("qrc:/qt/qml/com/showcase/qml/main.qml"));
engine.load(url);
```

**Qt 6.5+ Standard** (according to `simple-project.md` skill):
```cpp
engine.loadFromModule("com.showcase", "Main");  // ← preferred
```

**Reasoning**: `loadFromModule()` uses direct URIs and works with both file-system imports (dev) and resources (shipped binary). The `qrc:/` URL couples the loader to the resource prefix and filename.
